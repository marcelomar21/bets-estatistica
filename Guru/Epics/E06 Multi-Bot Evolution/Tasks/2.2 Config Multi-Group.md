# 2.2 Refatorar config.js para multi-group

## Objetivo

Transformar o config flat baseado em env vars em um sistema que carrega configs per-group do banco de dados, mantendo backward-compat.

## Action

1. Manter config global (supabase, APIs, retry, LLM)
2. Adicionar `loadGroupConfigs()` que le da tabela `bot_pool` (source of truth para token/IDs) JOIN `groups` (config per-group) no startup
3. Cada grupo carrega: `bot_pool.bot_token`, `bot_pool.admin_group_id`, `bot_pool.public_group_id` (do bot_pool), `postingSchedule`, `copyToneConfig`, `maxActiveBets` (do groups)
4. Manter env vars existentes como fallback (backward-compat para deploy atual com 1 bot)

## Files

| Arquivo | Descricao |
|---------|-----------|
| `lib/config.js` | Config flat via env vars -> DB-loaded per-group + env vars como fallback |

## Acceptance Criteria

- [ ] **AC 2.3**: Given `loadGroupConfigs()` chamado, when a tabela `groups` tem `copy_tone_config` e `max_active_bets` populados, then cada config e carregada no `BotContext.groupConfig`.
- [ ] `validateConfig()` aceita: ou env vars tradicionais (1 bot) ou configs carregadas do DB (N bots)

## Notes

- `validateConfig()` passa a aceitar: ou env vars tradicionais (1 bot) ou configs carregadas do DB (N bots).
- Config flat via env vars em `lib/config.js` precisa virar DB-loaded per-group.
- `bot_pool` e a source of truth para tokens — ao construir `BotContext`, ler de `bot_pool` (nao de `groups.bot_token`).
