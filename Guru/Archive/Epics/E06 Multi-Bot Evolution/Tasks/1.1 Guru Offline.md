# 1.1 Diagnosticar e corrigir Guru offline (B1 + B2)

## Objetivo

Restaurar o funcionamento do bot Guru da Bet que esta offline — nao faz disparos automaticos nem responde comandos no Telegram.

## Action

1. Verificar logs do servico Guru no Render (`srv-d5hp23a4d50c7397o1q0`)
2. Checar `getWebhookInfo` via API Telegram
3. Validar env vars (`TELEGRAM_BOT_TOKEN`, `GROUP_ID`, `BOT_MODE`, `TELEGRAM_ADMIN_GROUP_ID`)
4. Comparar config do Guru vs Osmar
5. Se webhook desconfigurado, re-registrar
6. Se bot crashando, identificar erro nos logs e corrigir

## Files

| Arquivo | Descricao |
|---------|-----------|
| Render dashboard (`srv-d5hp23a4d50c7397o1q0`) | Servico do bot Guru — logs, env vars, status |
| `bot/server.js` | Entry point do bot — webhook setup, scheduler bootstrap |

## Acceptance Criteria

- [ ] **AC 1.1**: Given bot Guru no Render, when o servico esta rodando, then o bot responde ao comando `/status` no grupo admin do Guru em menos de 5 segundos.
- [ ] **AC 1.2**: Given bot Guru configurado com posting_schedule enabled, when chega o horario configurado (10h/15h/22h), then o bot distribui e posta automaticamente no grupo publico do Guru.

## Notes

- Pode ser fix de config sem mudanca de codigo. Osmar funciona = codigo ok, problema e infra/config.
- **Se diagnostico revelar bug de codigo** (ex: race condition no startup, env var lida antes de inicializar, crash no `processWebhookUpdate`): investigar stack trace nos logs do Render, corrigir no codigo e fazer deploy.
- Se for crash silencioso sem log, adicionar `process.on('uncaughtException')` e `process.on('unhandledRejection')` com logging + restart graceful em `bot/server.js`.
