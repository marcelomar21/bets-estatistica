# 5.1 Refatorar server.js para multi-webhook

## Objetivo

Modificar o entry point do bot para suportar multiplos bots no mesmo processo Express, com uma rota de webhook por bot token.

## Action

1. Em `start()`: chamar `initBots()` (plural), receber array de `BotContext`
2. Para cada bot: registrar `app.post('/webhook/<token>', handler)` com closure sobre o `botCtx`
3. `processWebhookUpdate(update, botCtx)` — todas comparacoes de chat ID usam `botCtx.adminGroupId`
4. Registrar webhooks no Telegram para cada bot token
5. Remover `cachedGroupChatId` singleton — usar `botCtx.publicGroupId`

## Files

| Arquivo | Descricao |
|---------|-----------|
| `bot/server.js` | Entry point — webhook singleton -> multi-bot registry com N rotas |

## Acceptance Criteria

- [ ] **AC 5.1**: Given 1 processo Node.js rodando no Render, when mensagens chegam de ambos os grupos (Guru e Osmar), then cada mensagem e roteada para o handler correto do bot correspondente.
- [ ] **AC 5.4**: Given o processo unificado e deployado, when ambos os bots sao verificados, then `getWebhookInfo` retorna URLs corretas para cada token e ambos respondem a `/status`.

## Notes

- O Express app continua sendo 1 so. Multiplas rotas coexistem.
- URL do webhook: `POST /webhook/<BOT_TOKEN>` — 1 rota Express por token.
- `processWebhookUpdate()` faz comparacao direta de `msg.chat.id === config.telegram.adminGroupId` para decidir se e mensagem admin — precisa mudar para `botCtx.adminGroupId`.
- Nao existe dispatch table nem bot registry — tudo e if/else contra config estatica (estado atual).
