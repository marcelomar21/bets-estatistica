# Database Schema

PostgreSQL hosted on Supabase (project ref: `vqrcuttvcgmozabsqqja`, East US / North Virginia). All tables use RLS (Row Level Security) for multi-tenant isolation via `group_id`.

## Core Tables

### `suggested_bets`

Main bets table. Each row represents a single bet suggestion generated by the AI pipeline.

| Column | Type | Description |
|---|---|---|
| `id` | SERIAL PK | Auto-increment ID |
| `match_id` | INTEGER FK | References `league_matches.match_id` |
| `bet_market` | TEXT | Market type (e.g., "Over 2.5 Gols", "BTTS - Sim") |
| `bet_pick` | TEXT | Specific pick within the market |
| `odds` | NUMERIC | Current odds from pipeline |
| `odds_at_post` | NUMERIC | Odds at the time of posting to Telegram |
| `confidence` | NUMERIC | AI confidence score |
| `reasoning` | TEXT | AI reasoning/analysis text |
| `risk_level` | TEXT | Risk classification |
| `bet_category` | TEXT | Category: 'SAFE', etc. |
| `bet_status` | TEXT | Status: `generated`, `pending_link`, `pending_odds`, `ready`, `posted` |
| `bet_result` | TEXT | Result: `pending`, `success`, `failure`, `unknown`, `cancelled` |
| `result_reason` | TEXT | LLM justification for result |
| `result_updated_at` | TIMESTAMPTZ | When result was last evaluated |
| `eligible` | BOOLEAN | Legacy eligibility flag |
| `elegibilidade` | TEXT | Current eligibility: `elegivel`, `nao_elegivel`, `removida` |
| `promovida_manual` | BOOLEAN | Manually promoted (bypasses MIN_ODDS check) |
| `deep_link` | TEXT | Betting house deep link |
| `telegram_posted_at` | TIMESTAMPTZ | When posted to Telegram |
| `telegram_message_id` | BIGINT | Telegram message ID |
| `group_id` | UUID FK | References `groups.id` (multi-tenant) |
| `distributed_at` | TIMESTAMPTZ | When assigned to a group by distribution |
| `post_now_requested_at` | TIMESTAMPTZ | Manual post flag (on `groups`, not here) |
| `created_at` | TIMESTAMPTZ | Row creation time |

**Status transitions:**
```
generated -> pending_link -> ready -> posted
generated -> pending_odds -> ready -> posted
```

**Result transitions:**
```
pending -> success | failure | unknown | cancelled
```

| Column | Type | Description |

| `home_team_name` | TEXT | Home team name |
| `away_team_name` | TEXT | Away team name |
| `kickoff_time` | TIMESTAMPTZ | Match start time |

### `bot_pool`

Source of truth for Telegram bot tokens and chat IDs. One row per bot-group mapping.

| Column | Type | Description |
|---|---|---|
| `id` | SERIAL PK | Auto-increment |
| `group_id` | UUID FK NOT NULL | References `groups.id` |
| `bot_token` | TEXT NOT NULL | Telegram bot token |
| `is_active` | BOOLEAN | Whether this bot is active |
| `created_at` | TIMESTAMPTZ | Creation time |

**Planned new columns** (spec Task 2.3, migration 029):
- `admin_group_id` (BIGINT NOT NULL) -- Telegram admin group chat ID
- `public_group_id` (BIGINT NOT NULL) -- Telegram public group chat ID

The `bot_pool` table is the **single source of truth** for tokens and chat IDs. The `groups.bot_token` column is deprecated.

## RLS Policies

All queries from the bot use the Supabase `service_role` key (bypasses RLS). The admin panel uses the `anon` key with RLS policies enforced.

Multi-tenant isolation is achieved via:
- `group_id` column on `suggested_bets`, `members`, and other tenant-scoped tables
- Middleware in the admin panel (`createApiHandler`) injects `groupFilter` based on the authenticated user's role and assigned group
